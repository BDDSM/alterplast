Используя Расширение(Продление) DLL, База данных/OLE/Гнезда в Правильном DLL

Домой |  Краткий обзор |  Как Делают меня |  Страница популярных вопросов |  Подробности |  Выборка

При использовании расширения(продления) DLL от правильного DLL, если расширение(продление) DLL не телеграфировано(не связано) в CDynLinkLibrary цепочку object правильного DLL, Вы можете столкнуться один или больше набора связанных проблем. Поскольку отладочные версии MFC Базы данных, OLE, и Гнезд поддерживают DLLs, осуществлены как расширение(продление) DLLs, Вы могли бы видеть подобные проблемы, если Вы используете эти особенности MFC, даже если Вы явно не используете ни одного вашего собственного расширения(продления) DLLs. Немного признаков:

  • При попытке де-преобразовать в последовательную форму object типа класса, определенного в расширении(продлении) DLL, сообщение "Предупреждение: Не может загрузить CYourClass от архива. Класс, не определенный. " появляется в окне отладки СЛЕДА, и object будет не в состоянии преобразовывать в последовательную форму.

     
  • Исключение, указывающее 'плохой класс может быть брошено.

     
  • Ресурсы, сохраненные в расширении(продлении) DLL будут не в состоянии загружаться, потому что AfxFindResourceHandle возвращает ПУСТОЙ УКАЗАТЕЛЬ или неправильный маркер(дескриптор) ресурса.

     
  • DllGetClassObject, DllCanUnloadNow, и UpdateRegistry, Отменяют, RevokeAll, и RegisterAll функции члена COleObjectFactory будут не в состоянии определять местонахождение фабрики класса, определенной в расширении(продлении) DLL.

     
  • AfxDoForAllClasses не работает ни для каких классов в расширении(продлении) DLL.

     
  • Стандарт MFC база данных, гнезда, или ресурсы OLE будет не в состоянии загружаться. Например, AfxLoadString (AFX_IDP_SQL_CONNECT_FAIL) возвращает пустую строку, даже когда правильный DLL должным образом использует MFC классы Базы данных.

Решение этих проблем состоит в том, чтобы создать и экспортировать функцию инициализации в расширении(продлении) DLL, который создает CDynLinkLibrary object. Назовите(вызовите) эту функцию инициализации точно однажды от каждого правильного DLL, который использует расширение(продление) DLL.

MFC OLE, MFC База данных (или DAO), или MFC Поддержка Гнезд

Если Вы используете любое MFC OLE, MFC База данных (или DAO), или MFC поддержка Гнезд в вашем правильном DLL, соответственно, расширение(продление) отладки MFC DLLs MFCOxxD.DLL, MFCDxxD.DLL, и MFCNxxD.DLL (где обман - номер версии), связаны в автоматически. Вы должны назвать(вызвать) предопределенную функцию инициализации для каждого из  этих DLLs, которые Вы используете.

Для поддержки базы данных, добавьте запрос к AfxDbInitModule к CWinApp вашего правильного DLL:: функция InitInstance. Удостоверьтесь, что этот запрос происходит перед любым запросом базового класса или любым добавленным кодом, который обращается к MFCDxxD.DLL. Эта функция не берет никаких параметров и возвращается пусто.

Для поддержки OLE, добавьте запрос к AfxOleInitModule к CWinApp вашего правильного DLL:: InitInstance. Обратите внимание, что COleControlModule функция InitInstance называет(вызывает) AfxOleInitModule уже, так если Вы строите OLE, управляют и используют COleControlModule, Вы не должны добавить этот запрос к AfxOleInitModule.

Для поддержки Гнезд, добавьте запрос к AfxNetInitModule к CWinApp вашего правильного DLL:: InitInstance.

Обратите внимание, что компоновки выпуска MFC DLLs и приложений не используют отдельный DLLs для базы данных, гнезд, или поддержки OLE. Однако, безопасно назвать(вызвать) эти функции инициализации в режиме выпуска.

CDynLinkLibrary Объекты

В течение каждой из операций, упомянутых в начале этой статьи(изделия), MFC должен искать желательное значение или object. Например, в течение де-преобразования в последовательную форму, MFC должен перерыть все в настоящее время доступные классы во время выполнения, чтобы соответствовать объектам в архиве с их надлежащим классом во время выполнения.

Поскольку часть этих исследований, MFC просматривает через все расширение(продление) DLLs в использовании, идя цепочка объектов CDynLinkLibrary. CDynLinkLibrary объекты присоединяются автоматически к цепочке в течение их конструктора и созданы каждым расширением(продлением) DLL в свою очередь в течение инициализации. Кроме того, каждый модуль (прикладной или правильный DLL) имеет его собственную цепочку объектов CDynLinkLibrary.

Для расширения(продления) DLL, чтобы телеграфироваться(связаться) в CDynLinkLibrary цепочку, это должно создать CDynLinkLibrary object в контексте каждого модуля что использования расширение(продление) DLL. Поэтому, если расширение(продление) DLL собирается использоваться от правильного DLLs, это должно обеспечить экспортируемую функцию инициализации, которая создает CDynLinkLibrary object. Каждый правильный DLL, что использования расширение(продление) DLL должны назвать(вызвать) экспортируемую функцию инициализации.

Если расширение(продление) DLL только собирается использоваться от приложения MFC (.EXE) и никогда от правильного DLL, то достаточно создать CDynLinkLibrary object в DllMain DLL's расширения(продления). Это - то, что AppWizard расширение(продление) DLL код делает. При загрузке расширения(продления) DLL неявно, DllMain загружается и выполняется прежде, чем приложение когда-либо начинается. Любые CDynLinkLibrary создания телеграфированы(связаны) в заданную по умолчанию цепочку, которую MFC DLL резервирует для приложения MFC.

Обратите внимание, что это - плохая идея, чтобы иметь множественные объекты CDynLinkLibrary от одного расширения(продления) DLL в любом цепочка, особенно если расширение(продление) DLL будет динамически разгружено от памяти. Назовите функцию инициализации не раз от любого модулем.

Типовой Код

Этот типовой код предполагает, что правильный DLL неявно связывается с расширением(продлением) DLL. Это достигнуто, связываясь с библиотекой импорта (.LIB) расширения(продления) DLL при строительстве(формировании) правильного DLL.

Эти строки должны быть в источнике расширения(продления) DLL:

// YourExtDLL.cpp:

// стандарт MFC расширение(продление) DLL подпрограммы
*include "afxdllx.h"

статический AFX_EXTENSION_MODULE ОКОЛО extensionDLL = {ПУСТОЙ УКАЗАТЕЛЬ, ПУСТОЙ УКАЗАТЕЛЬ};

экскрачка "C" int APIENTRY
DllMain (HINSTANCE hInstance, DWORD dwReason, LPVOID lpReserved)
{если
    (dwReason == DLL_PROCESS_ATTACH)
    {//
        расширение(продление) DLL одноразовая(бывшая) инициализация.
        если (! AfxInitExtensionModule (extensionDLL, hInstance))
           возвращаются 0;}
    возвращаются
    1; // хорошо}

// Экспортирован DLL инициализация, работают в
контексте // прикладной или правильной DLL
экскрачки "C" пусто WINAPI InitYourExtDLL ()
{//,
    создают новый CDynLinkLibrary для этого приложения
    новый CDynLinkLibrary (extensionDLL);

    // добавить другую инициализацию здесь}

Убедитесь, что экспортировали функцию InitYourExtDLL. Это могло быть сделано, используя __ declspec (dllexport), или в .DEF файле вашего DLL следующим образом:

// YourExtDLL.Def:
БИБЛИОТЕКА      YOUREXTDLL
ЗАКОДИРУЕТ         ПРЕДВАРИТЕЛЬНУЮ ЗАГРУЗКУ ПОДВИЖНАЯ НЕЗАЩИЩЕННАЯ
ПРЕДВАРИТЕЛЬНАЯ ЗАГРУЗКА ДАННЫХ         ЕДИНСТВЕННЫЙ(ОТДЕЛЬНЫЙ)
ЭКСПОРТ
    InitYourExtDLL

Добавьте запрос к InitInstance члену CWINAPP-ПОЛУЧЕННОГО object в каждом правильном DLL использование расширения(продления) DLL: 

// YourRegularDLL.cpp:

класс CYourRegularDLL: публика CWinApp
{публика:
виртуальный
    BOOL InitInstance (); // Инициализация
    виртуальный int ExitInstance (); // Завершение

    // ничто специальное для конструктора
    CYourRegularDLL (LPCTSTR pszAppName): CWinApp (pszAppName) {}};

BOOL CYourRegularDLL:: InitInstance ()
{//
  любая DLL инициализация идет сюда
    TRACE0 ("ВАШ правильный DLL initializing\n");
